# 决策记录：第一阶段聚焦富途港股全量交易数据

## 标题

- 决策标题：第一阶段以富途港股全量交易数据跑通端到端方案
- 日期：2026-06-23
- 状态：已通过

## 背景与上下文

- 背景：个人投资数据系统长期目标是覆盖多平台、多账户、多资产类别，但第一阶段如果直接铺开全品类，数据库 schema、Base 镜像、导入解析、收益核算和税务口径都会同时膨胀。
- 约束：已有飞书 Base Demo 主要围绕港股、现金流水、IPO/打新、FIFO 已实现收益等场景，且用户希望先用一个真实数据范围跑通整体流程。
- 驱动因素（为什么必须做选择）：第一阶段需要一个足够真实、足够复杂、但边界可控的样本域，用来验证数据库主源、Base 单向镜像、Lot/Allocation、费用口径、收益核算、追溯和对账链路。

## 目标

- 我们希望达成什么：先用富途港股相关的全量可得交易数据跑通端到端流程，形成可验证的数据库 schema、Base 镜像、导入映射、收益计算和对账方法。
- 成功指标（可选）：富途港股样本数据能够完整落入数据库；核心数据可单向镜像到 Base；买卖、现金、费用、IPO、分红/利息等关键事实可追溯；FIFO 已实现收益可计算并解释。

## 备选方案

### 方案 A：第一阶段覆盖全资产全券商

- 描述：一开始就把股票、基金、债券、期权、加密资产、多个券商和多税务场景全部纳入第一阶段。
- 优点：长期愿景完整，不需要后续大规模扩展。
- 缺点：范围过大，难以快速验证；容易为了未验证场景提前复杂化 schema。
- 成本/依赖：需要多个券商结单样本、不同资产处理规则、多币种和税务口径。
- 风险：第一版迟迟无法形成可用闭环。

### 方案 B：第一阶段聚焦富途港股全量交易数据

- 描述：先用富途港股相关的可得交易事实作为主路径，覆盖成交、资金流水、IPO/打新、费用、税费、融资利息、分红、利息、转入转出、可能的公司行动等数据，并跑通数据库到 Base 的单向镜像。
- 优点：真实度足够高，复杂度可控；与已有 Demo 贴近；能覆盖 IPO、费用去重、FIFO、现金对账等关键难点。
- 缺点：不能直接证明其他券商和非港股资产的适配性。
- 成本/依赖：需要确认富途导出格式、字段覆盖、历史数据完整性和样本脱敏策略。
- 风险：如果过度贴合富途字段，后续扩展到其他券商时可能需要重构。

### 方案 C：先做抽象 schema，不绑定真实券商样本

- 描述：先基于草案做通用数据库模型和 Base 模板，不接入具体券商数据。
- 优点：抽象上更干净，不受单个券商格式影响。
- 缺点：缺少真实数据压力，容易忽略导入、对账和异常处理细节。
- 成本/依赖：需要后续再补真实样本验证。
- 风险：设计看起来完整，但一遇到真实结单就需要返工。

## 决策

- 选择了什么：采用方案 B。第一阶段聚焦富途港股的所有可得交易数据，用这一范围跑通整个流程和方案。
- 为什么：富途港股样本既贴近已有 Demo，又包含交易、现金、IPO/打新、费用、融资利息、分红等复杂场景，适合检验系统最核心的事实建模、成本批次、收益计算和对账能力。
- 放弃了什么（以及原因）：不在第一阶段端到端覆盖其他券商和非港股资产，因为会稀释验证重点；也不先做纯抽象 schema，因为缺少真实数据压力。
