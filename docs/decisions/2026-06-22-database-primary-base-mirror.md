# 决策记录：数据库主源 + 飞书 Base 镜像

## 标题

- 决策标题：第一版以数据库为 标准真相源，同时兼容飞书 Base 模板和单向可读镜像
- 日期：2026-06-22
- 状态：已通过

## 背景与上下文

- 背景：个人投资数据系统需要统一承接多平台、多账户、多资产类别的数据，并支持收益核算、税务准备、账户对账和复盘追踪。已有飞书 Base Demo 证明 Base 适合阅读、协作和轻量验证，但长期计算、约束、一致性、追溯和自动化更适合由数据库承载。
- 约束：数据会涉及个人账户和交易信息；系统需要保留可追溯事实、避免重复统计，并在未来支持多币种、多资产类别和券商结单解析。
- 驱动因素（为什么必须做选择）：第一版要确定 标准真相源，否则数据库、飞书 Base、Excel/CSV 和后续解析器之间会出现多头维护和口径漂移。

## 目标

- 我们希望达成什么：用数据库承载稳定 schema、约束、计算和长期演进，同时保留飞书 Base 的可读性、协作性和模板化便利；第一阶段同步方向为数据库 -> Base，不允许 Base 回写数据库。
- 成功指标（可选）：同一批样例数据能够从数据库稳定同步到 Base；核心收益计算以数据库结果为准；Base 可用于人工阅读、校验和协作备注，但不产生 canonical 数据变更。

## 备选方案

### 方案 A：飞书 Base 作为主源

- 描述：以 Base 为主要工作台和数据真相源，利用多维表字段、关联、lookup、formula 和 dashboard 完成记录与统计。
- 优点：上手快、可读性强、适合人工校验和协作；已有 Demo 可直接延续。
- 缺点：复杂约束、批量导入、版本迁移、计算审计、自动化解析和多币种/税务复杂逻辑会逐渐变重。
- 成本/依赖：依赖飞书 Base 能力、权限、公式维护和人工表结构治理。
- 风险：公式和 lookup 逻辑长期堆积后难以测试；多份视图和手工编辑可能导致口径漂移。

### 方案 B：数据库作为主源，飞书 Base 作为镜像

- 描述：以数据库作为 标准真相源，设计稳定 schema、约束、计算和同步协议；飞书 Base 作为可读、协作、展示友好的模板和同步镜像。
- 优点：长期一致性、可测试性、迁移能力和自动化解析能力更强；Base 仍保留易读和协作优势。
- 缺点：第一版需要额外设计字段映射、同步方向、冲突处理和脱敏策略。
- 成本/依赖：需要数据库 schema、同步脚本/流程、Base 模板字段映射和一致性校验。
- 风险：单向镜像会限制 Base 侧协作编辑能力；如果未来开放回写，需要重新设计字段所有权、冲突处理和审计。

### 方案 C：只做文件/Excel/CSV 模板

- 描述：先不建立数据库或 Base 主工作台，只定义 CSV/Excel 文件模板和手工处理流程。
- 优点：成本最低，便于快速起步和导入历史数据。
- 缺点：难以表达关联、约束、成本批次、同步、审计和多视图计算。
- 成本/依赖：需要较强手工规范和后续迁移工作。
- 风险：一旦历史数据量和资产类别增加，文件模板会很快变成迁移负担。

## 决策

- 选择了什么：采用方案 B。第一版先做成数据库，同时兼容飞书 Base 模板模式；数据从数据库单向同步到 Base，以提高可读性和协作体验。Base 第一阶段不作为写入口，不回写数据库。
- 为什么：数据库更适合承担稳定 schema、约束、计算、追溯和自动化；飞书 Base 更适合阅读、校验、协作和轻量展示。两者结合能兼顾长期正确性和日常可用性。
- 放弃了什么（以及原因）：不让飞书 Base 单独承担 标准真相源，因为复杂收益、税务、同步和解析逻辑会逐步超出表格公式的舒适区；也不只做 CSV/Excel 模板，因为它不足以支撑长期关联和审计。

## 复盘触发条件

- Base 镜像和数据库出现无法自动解释的数据差异。
- Base 侧协作编辑需求变强，需要从单向镜像改为可回写。
- 数据库 schema 变更导致 Base 模板频繁破坏或维护成本过高。
- 税务/收益计算逻辑需要由 Base 直接展示明细，而当前同步粒度不足。
